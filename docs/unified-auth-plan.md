# Единая точка входа для разных платформ счетчиков

## Summary
Сделать отдельный Node.js сервис с веб-UI и HTTP API, который принимает `login/password`, параллельно проверяет их во всех поддерживаемых платформах, определяет успешные совпадения, при необходимости дает пользователю выбрать платформу, после чего создает внутреннюю сессию и дальше сам работает как прокси к внешнему API, отдавая данные счетчиков в едином формате.

Текущий репозиторий содержит UDP-приемник RTU102, поэтому эту задачу лучше реализовывать как новый подсервис или отдельный пакет в этом же репозитории, а не встраивать в текущий `node_receiver`.

## Key Changes

### 1. Архитектура сервиса
- Выделить 4 слоя: `web-ui`, `auth-orchestrator`, `platform adapters`, `meter-data normalizer`.
- `auth-orchestrator` запускает проверки по всем платформам параллельно с общим таймаутом и ограничением конкуренции.
- Для каждой платформы реализовать адаптер с единым контрактом:
  - `probeCredentials(login, password): ProbeResult`
  - `createSession(login, password): PlatformSession`
  - `fetchMeters(session): UnifiedMeter[]`
  - `fetchReadings(session, filters?): UnifiedReading[]`
- Все специфичные URL, таймауты, client secrets и заголовки держать в конфиге + `env`, не в коде.
- Внутреннюю пользовательскую сессию делать собственной, а токены/куки внешней платформы хранить как часть серверной прокси-сессии.

### 2. Поведение логина
- `POST /auth/login` принимает `login/password`.
- Сервис одновременно вызывает `probeCredentials` у всех адаптеров.
- Если совпадений нет: возвращать единое сообщение `invalid credentials`.
- Если совпадение одно: сразу создавать прокси-сессию и логинить пользователя.
- Если совпадений несколько: возвращать список платформ-кандидатов для выбора в UI.
- `POST /auth/select-platform` завершает логин после выбора платформы пользователем.
- Внутренняя сессия должна хранить:
  - `userId` или хеш логина
  - `platformId`
  - `platformSessionRef`
  - срок жизни сессии
- По умолчанию выбрать stateful-сессию на сервере: `Redis` для сессий, fallback на in-memory только для локальной разработки.

### 3. Унификация данных
- Ввести единые DTO:
  - `UnifiedMeter { id, name, serialNumber, platformId, status, address?, tags? }`
  - `UnifiedReading { meterId, timestamp, value, unit, type, quality?, sourcePlatform }`
- Каждый адаптер маппит внешний ответ в эти DTO.
- Дополнительно сохранять сырой ответ платформы в debug/audit-логе только по флагу, чтобы упрощать разбор интеграций без утечки лишних данных в обычный ответ.
- Основные API после логина:
  - `GET /me`
  - `GET /meters`
  - `GET /meters/:id/readings?from=&to=`
  - `POST /auth/logout`

### 4. Безопасность и отказоустойчивость
- Не логировать пароль и полные внешние токены.
- Добавить rate limit на `POST /auth/login` и простой lockout/backoff по IP + логину.
- Для платформ использовать короткие таймауты и circuit breaker/retry только для сетевых ошибок, не для `401/403`.
- Результаты `probeCredentials` нормализовать в статусы: `success | invalid_credentials | unavailable | timeout | error`.
- Если платформа недоступна, это не должно ломать логин через другие платформы; недоступные источники исключаются из выбора.

## Public Interfaces / Types
- HTTP API:
  - `POST /auth/login`
  - `POST /auth/select-platform`
  - `POST /auth/logout`
  - `GET /me`
  - `GET /meters`
  - `GET /meters/:id/readings`
- Основные доменные интерфейсы:
  - `PlatformAdapter`
  - `ProbeResult`
  - `PlatformSession`
  - `UnifiedMeter`
  - `UnifiedReading`
- Конфиг платформ:
  - `platformId`
  - `baseUrl`
  - `timeouts`
  - `authMode`
  - `client credentials/env key names`
  - `enabled`

## Test Plan
- Успешный логин, когда подходит ровно одна платформа.
- Неуспешный логин, когда ни одна платформа не принимает учетные данные.
- Конфликт, когда одинаковые `login/password` подходят нескольким платформам, и UI предлагает выбор.
- Частичная деградация: одна платформа отвечает, другая в timeout, третья недоступна.
- Нормализация данных: разные внешние форматы маппятся в одинаковые `UnifiedMeter` и `UnifiedReading`.
- Истечение внутренней сессии и повторная авторизация.
- Защита от brute force/rate limiting.
- Контрактные тесты для каждого адаптера на типовые ответы `200/401/403/5xx/timeout`.

## Assumptions
- Реализуется новый сервис рядом с текущим RTU102-кодом, а не изменение существующего UDP receiver.
- Платформ немного и их список известен заранее, поэтому конфиг + `env` достаточно; админка для управления платформами не нужна на первом этапе.
- После логина сервис сам ходит во внешнюю платформу и отдает пользователю уже унифицированные данные.
- При множественном совпадении окончательный выбор платформы делает пользователь.
- Базовый стек по умолчанию: `Node.js + TypeScript + Fastify/Nest-like HTTP layer + Redis sessions + server-rendered UI или легкий SPA`.
